# 通用前端开发流程

## 概述

帮助用户从零开始完成前端项目的全生命周期：需求探测 → 需求文档 → 技术选型 → 编码实现 → 测试修复 → 上线发布。
适用于 HTML/CSS/JS、React、Vue、小程序等各类前端项目。

## 触发场景

当用户的请求包含以下内容时启用：
- 开始一个新项目（"帮我做一个…""写一个…""创建一个…"）
- 讨论开发流程或项目规划
- 询问技术选型或架构设计
- 要求测试或修复 bug
- 准备上线或部署

---

## 使用方法

### 第一步：需求探测（Phase 0，必须先做）

**核心原则：每次只问 3-4 个问题，不要一次性抛出太多。**

在写代码之前，按以下四轮顺序提问。即使用户的需求看起来已经很详细，也必须至少完成前两轮。

#### 第一轮：项目基础（最先问）

| # | 问题 | 目的 |
|---|------|------|
| 1 | 用一两句话描述一下这个项目是什么？ | 确定项目范围和核心目的 |
| 2 | 给谁用？解决他们什么问题？ | 决定 UX 设计和功能优先级 |
| 3 | 有没有参考的产品或竞品？你喜欢/不喜欢它什么地方？ | 快速对齐设计方向 |

#### 第二轮：范围与功能（用户回答第一轮后问）

| # | 问题 | 目的 |
|---|------|------|
| 4 | 最核心的 2-4 个功能是什么？哪些可以留到后面版本？ | 防止范围蔓延，定义 MVP |
| 5 | 用户需要注册登录吗？ | 确定认证复杂度 |
| 6 | 数据需要跨会话或跨设备保存吗？ | 决定存储方案（localStorage / 数据库 / 云端） |
| 7 | 使用规模多大？（单人、小组、公开） | 影响架构和性能规划 |
| 8 | 有哪些页面/界面？列出用户能看到的每个视图。 | 建立页面清单 |
| 9 | 这些页面之间怎么跳转？画一下导航流程。 | 提前暴露导航问题 |

> 如果用户对 8-9 不确定，可以主动提供一个页面树草图，例如：
> ```
> 首页 → 登录/注册
>      → 主界面（底部标签：动态、搜索、我的）
>           ├── 动态 → 详情页
>           ├── 搜索 → 搜索结果
>           └── 我的 → 设置
> ```

#### 第三轮：技术偏好（根据情况接着问）

| # | 问题 | 目的 |
|---|------|------|
| 10 | 技术栈有偏好吗？（原生 HTML/CSS/JS、React、Vue 等）不确定可以说。 | 决定架构和工具链 |
| 11 | 设计风格有偏好吗？（极简、活泼、专业、暗色模式、移动优先） | 设定视觉方向 |
| 12 | 有什么平台或浏览器限制？（必须离线可用、仅移动端、微信小程序等） | 识别非功能性需求 |
| 13 | 有偏好的部署目标吗？（GitHub Pages、云平台、自建服务器） | 影响构建和部署流程 |

#### 第四轮：边界情况与特殊需求（按需问）

| # | 问题 | 目的 |
|---|------|------|
| 14 | 出问题时怎么办？（断网、接口失败、空数据） | 定义错误处理和兜底策略 |
| 15 | 需要多语言（i18n）或无障碍（a11y）支持吗？ | 从一开始影响组件设计 |
| 16 | 需要对接第三方 API 或服务吗？（AI、地图、支付、统计） | 提前发现集成复杂度 |

#### 何时停止提问

以下情况可以直接进入开发：
- 用户明确说"直接开始吧"或"我相信你的判断"
- 已有足够信息定义 MVP 范围
- 至少完成了前两轮提问

如果用户无法回答某些问题，做出合理假设，明确告知用户，然后继续推进。

### 第二步：判断项目阶段

根据用户当前状态，确定从哪个阶段开始：

| 用户说… | 从哪个阶段开始 |
|---------|-------------|
| "我有个想法…""帮我做一个…" | Phase 1：需求文档 |
| "我有设计稿/需求文档""选什么技术栈" | Phase 2：技术选型与架构 |
| "开始写代码""实现这个功能" | Phase 3：编码实现 |
| "帮我测试""找找 bug" | Phase 4：测试修复 |
| "部署""上线""发布" | Phase 5：上线发布 |

---

### 第三步：按阶段推进

#### Phase 1：需求文档

1. **产品需求文档（PRD）**：一句话定位、目标用户、核心功能模块、非功能性需求
2. **数据结构设计**：核心实体、字段名/类型/含义、实体关系、附带示例数据
3. **算法和规则**：核心计算逻辑用伪代码或自然语言描述，注明边界情况
4. **视觉设计规范**：配色方案、字体层级、组件样式约定（圆角、阴影、间距）

#### Phase 2：技术选型与架构

1. **技术选型**：
   - 轻量离线应用 → HTML + Vanilla JS
   - 复杂交互/路由 → React / Vue + UI 库
   - 持久化/多用户 → 前端 + 后端 + 数据库

2. **架构设计**：模块关系图、状态管理方案、页面/路由切换逻辑、持久化策略

#### Phase 3：编码实现（按顺序）

1. **骨架先行**——HTML/CSS 布局框架
2. **状态管理**——全局状态对象和场景切换函数
3. **数据注入**——将文档中的数据转为代码常量
4. **逐页面实现**——启动页 → 主界面 → 核心功能 → 结果页
5. **弹窗和辅助模块**——设置、存档、提示等覆盖层
6. **持久化**——保存/加载逻辑
7. **音效和动画**——最后润色

**编码原则**：
- 每个函数单一职责
- 弹窗独立渲染，不耦合特定页面
- UI 交互统一注册，避免分散的点击处理
- 数据与逻辑分离
- 关键流程用 try-catch 防止单点故障导致整个页面卡死

#### Phase 4：测试修复

**功能测试**：
- 正常路径：完整走完所有页面
- 异常路径：操作失败、中途退出、存档恢复
- 边界情况：空数据、极值

**代码审查清单**：
- JS 语法错误检查
- 中文引号/特殊字符检查
- console.log / debugger 残留清理
- 重复 ID / 重复调用检查
- 死代码检查
- 页面生命周期陷阱（导航/刷新时自动保存可能覆盖已清空的数据）

**常见陷阱**：

| 问题 | 表现 | 预防 |
|------|------|------|
| 弹窗与页面耦合 | 关闭弹窗跳到错误页面 | 弹窗独立于页面状态渲染 |
| 重复 ID | 后面注册的按钮不响应 | 每个交互元素使用唯一 ID |
| 状态转换遗漏 | 切换页面后上一页 UI 残留 | 切换时清理中间状态 |
| 自动保存写回 | 清空的数据重置后又出现 | 重置前设标记，跳过存盘 |
| CDN 缓存 | 部署后显示旧内容 | 文件名追加内容哈希 |

#### Phase 5：上线发布

1. **项目清理**：添加 `.gitignore`、写 `README.md`、确保文档与代码同步
2. **Git 工作流**：
   ```bash
   git init
   git add -A
   git commit -m "提交信息"
   git branch -M main
   git remote add origin <仓库地址>
   git push -u origin main
   ```
3. **可选步骤**：添加仓库描述和标签、开启 GitHub Pages 或云静态托管、配置 CDN 缓存策略

## 适用场景

这个流程适用于：
- 单页面 / 轻量级 Web 应用
- 小游戏和互动工具
- 原型验证项目
- 个人项目和演示作品

需要额外步骤的场景：
- 多人协作 → 分支策略 + 代码审查流程
- 后端服务 → API 设计 + 数据库迁移 + 运维
- 原生移动端 → 平台适配 + 应用商店审核

## 快速参考流程图

```
需求文档 → 数据设计 → 算法/视觉规范
    ↓
技术选型 + 架构设计
    ↓
骨架 → 状态管理 → 数据注入
    ↓
逐页面实现 → 弹窗/辅助 → 持久化
    ↓
全流程测试 → 代码审查 → Bug 修复
    ↓
README + .gitignore → Git 提交 → 推送上线
```
